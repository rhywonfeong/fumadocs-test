## 基本用法

在 CI 环境中，hpsetup 检测到 `CI` 环境变量后自动进入非交互模式并启用更新。只需一条命令：

```bash
HEROUI_KEY=hp_xxxx npx -y hpsetup@latest
```

不需要 `--update`、`--auto` 或任何其他 flag。

## GitHub Actions

将 `HEROUI_KEY` 添加到仓库的 Secrets 中，然后在 workflow 中使用：

```yaml
name: Install HeroUI Pro
on: [push, pull_request]

jobs:
  install:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npx -y hpsetup@latest
        env:
          HEROUI_KEY: ${{ secrets.HEROUI_KEY }}
```

## Vercel

在 Vercel 项目的 Settings → Build & Development Settings → Install Command 中设置：

```
npx -y hpsetup@latest
```

然后在 Environment Variables 中添加 `HEROUI_KEY`。

Vercel 的 Build 环境自带 `CI=true`，hpsetup 会自动检测。

## CI 行为详解

CI 模式启用后，以下机制确保安装可靠性：

### 1. 始终从 CDN 下载

CI 环境是干净的（无本地缓存、无制品），因此 hpsetup 在 CI 中**始终从 CDN 下载 tarball**，不检查 node_modules 或本地缓存。

具体行为：`process.env.CI` 检测后隐含 `--update`，即使已安装版本与 registry latest 一致，也会跳过"already on the latest version"的提前返回，直接进入下载流程。

这解决了 `--ignore-scripts` 安装 stub 包后制品目录为空、但版本号匹配导致 hpsetup 误判为"已最新"的问题。

### 2. stub 包检测

通过 `hasRealArtifacts` 检查 `index.js` / `index.mjs` 是否存在于产物目录中，而非仅检查目录是否存在。

npm 上的 stub 包只包含 `dist/postinstall/index.js`（空壳 postinstall 脚本），不含真实组件代码。此检测确保不会将 stub 包误认为已安装。

### 3. 下载失败 fallback

CDN 下载失败时不会直接退出，而是依次尝试：

```
CDN 下载 → 当前版本缓存 → 上一版本缓存 → 退出
```

### 4. 版本检查重试

对 npm registry 的请求使用 3 次递增超时重试（3s → 6s → 10s）。即使全部失败，也不会跳过更新——包会被标记为"版本未知"并强制走更新流程。

### 5. 版本不一致自动修复

在 `--auto` 模式下，如果 `package.json` 中声明的版本与 `node_modules` 中的实际版本不一致，hpsetup 会自动执行 `npm install @latest` 来同步两者。

## 最佳实践

- 始终使用 `npx -y hpsetup@latest`，不要锁定版本号
- 将 `HEROUI_KEY` 存储在 CI 平台的 Secrets 中，不要硬编码
- 不需要添加 `--update` 或 `--auto`，CI 环境会自动处理
